Introduction to Django

Creating a Virtual Environment

A virtual environment is a self-contained directory that contains a Python installation and a set of packages.
It allows you to manage dependencies for different projects separately. Create a new directory for your project. For example, "University_Platform", switch to that directory in a terminal, and create a virtual environment. 
# Create a virtual environment
python -m venv UniversityPlatformEnv 

# Activate the virtual environment
# On Windows:
UniversityPlatformEnv\Scripts\activate
# On macOS/Linux:
source UniversityPlatformEnv/bin/activate

Installing Django

With your virtual environment activated, install Django using pip, the Python package manager. 
pip install django

Creating a Django Project

Use the django-admin command-line tool to create a new Django project called University_Platform. 
django-admin startproject University_Platform .
The dot (.) at the end of the command tells Django to create the project in the current directory. If you look in your project directory, using `ls` (on macOS/Linux) or `dir` (on Windows), you should see a new folder called University_Platform containing several files.
  • manage.py: A command-line utility that lets you interact with this Django project in various ways.
  • University_Platform/ directory contains four files, the most important of which are settings.py, urls.py, and wsgi.py.
    • settings.py: Controls how Django interacts with your system and manages your project.
    • urls.py: Tells Django which pages to build in response to browser requests.
    • wsgi.py: Helps Django serve the files it creates. The filename is an acronym for web server gateway interface.

Creating the Database

Django uses a database to store information for your project. By default, Django is configured to use SQLite, a lightweight database that is easy to set up and requires no additional configuration. To create the database, run the following command: 
python manage.py migrate
Any time we modify a database, we say we are migrating the database. Issuing the migrate command for the first time tells Django to make sure the database matches the current state of the project. The first time we run this command in a new project using SQLite, Django will create a new database for us.
Running the ls command (dir on Windows) shows that Django created another file called db.sqlite3.

Running the Development Server

Django includes a lightweight web server for development purposes. To start the server, run the following command: 
python manage.py runserver
After starting the server, open a web browser and navigate to http://localhost:8000. You should see the default Django welcome page, indicating that your project is set up correctly.
To stop the server, return to the terminal and press Ctrl+C.
If you receive the error message That port is already in use, tell Django to use a different port by entering python manage.py runserver 8001 and cycle through higher numbers until you find an open port. For example: 
python manage.py runserver 8001

Starting a Django App

In Django, a project is a collection of settings and configurations for an entire website, while an app is a web application that does something specific, such as a blog or a forum. A project can contain multiple apps.
You should still be running runserver in the terminal window you opened earlier. Open a new terminal window (or tab) and navigate to the directory that contains manage.py. Activate the virtual environment, and then run the startapp command: 
python manage.py startapp Home
The command startapp appname tells Django to create the infrastructure needed to build an app. If you look in the project directory now, you'll see a new folder called Home. Open that folder to see what Django has created. The most important files are models.py, admin.py, and views.py. To include our app in the overall project, open settings.py (in the University_Platform/University_Platform directory), and you'll see a section that tells Django which apps are installed in the project: 
# Application definition
INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles',
    # My apps
    'Home',  # Add the Home app here
]
Add 'Home', to the INSTALLED_APPS list to include the Home app in the project.

Making Pages

Making web pages with Django consists of three stages: defining URLs, writing views, and writing templates.
First, you must define patterns for URLs. A URL pattern describes the way the URL is laid out and tells Django what to look for when matching a browser request with a site URL so it knows which page to return.
Each URL then maps to a particular view—the view function retrieves and processes the data needed for that page. The view function often calls a template, which builds a page that a browser can read.
In Django, views are Python functions that handle requests and return responses. Views are defined in the views.py file of your app.

Mapping a URL

To map a URL to a view, you need to define a URL pattern in the urls.py file of your app. We need to include the URLs for the TimeTable app in the project's main urls.py file located in the University_Platform/University_Platform directory. Open that file and modify it as follows: 
from django.contrib import admin
from django.urls import path, include
urlpatterns = [
    path('admin/', admin.site.urls),
    path("", include('Home.urls')),,  # Include Home app URLs
]
Next, create a new file called urls.py in the Home app directory and add the following code to define the URL patterns for the app: 
"""Defines URL patterns for Home app."""
from django.urls import path

from . import views
urlpatterns = [
    path('', views.index, name='index'),
    path('News/', views.news, name='News'),
    path('About_Us/', views.about_us, name='About_Us'),
    path('Contact_Us', views.contact_us, name='Contact_Us'),
]
In this example, we define three URL patterns:
- The root URL ('') maps to the index view.
- The URL 'News/' maps to the news view.
- The URL 'About_Us/' maps to the about_us view.
- The URL 'Contact_Us' maps to the contact_us view.
Each URL pattern is associated with a view function defined in views.py.

Writing Views

A view function takes in information from a request, prepares the data needed to generate a page, and then sends the data back to the browser, often by using a template that defines what the page will look like.
The file views.py in Home was generated automatically when we ran the command python manage.py startapp.
Here's an example of how to create a simple view that displays different pages for Home app: 
from django.shortcuts import render

# Create your views here.

def index(request):
    return render(request, 'Home/index.html')

def about_us(request):
    return render(request, 'Home/About_Us.html')

def contact_us(request):
    return render(request, 'Home/Contact_Us.html')

def news(request):
    return render(request, 'Home/News.html')
In this example, we define four view functions: index, about_us, contact_us, and news. Each function takes a request object as a parameter and uses the render shortcut to render a template and return an HTTP response.
The templates Home/index.html, Home/About_Us.html, Home/Contact_Us.html, and Home/News.html will be created later to define the structure of the corresponding web pages.

Template Inheritance

Templates define the structure and layout of the HTML pages that are sent to the browser. In Django, templates are stored in a templates directory within your app.
Create a new directory called templates inside the Home app directory, and then create another directory called Home inside the templates directory. This structure helps Django locate the templates for the Home app.
Template inheritance allows you to create a base template that defines the overall structure of your web pages, and then extend that base template in other templates to add specific content. This helps maintain a consistent look and feel across your website and reduces code duplication.
Let's create a base template called base_Home.html in the Home/templates/Home directory.
This base template will define the common header and footer for all pages in the Home app. The header will include a h1 tag and navigation bar with links to the course list, teacher list, and room list pages. The footer will include a simple copyright notice. 
<!DOCTYPE html>
<html>
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>University</title>
</head>
<body>
<header>
<nav class="navbar">
    <div class="nav-logo">
        <img src="logo.png" alt="Logo">
    </div>

    <ul class="nav-links">
<li><a href="{% url 'News' %}">News</a></li>
        <li><a href="{% url 'About_Us' %}">About Us</a></li>
            <li><a href="{% url 'Contact_Us' %}">Contact Us</a></li>

</ul>

    <div class="auth-links">
       
            <a href="{% url 'login_student' %}">Sign In</a>
        <span class="divider">|</span>
            <a href="{% url 'register_student' %}">Sign Up</a>

    </div>
</nav>
    </header>
                <main class="container">
                     {% block content %}
                    {% endblock %}
                </main>
    <footer class="footer">
    <div class="footer-container">

        <div class="footer-col">
            <h3>ENSIA</h3>
            <p>National School of Artificial Intelligence.
               Excellence in education, research, and innovation.</p>
        </div>
        <div class="footer-col">
            <h4>Quick Links</h4>
            <ul>
                <li><a href="#">Home</a></li>
                <li><a href="#">Courses</a></li>
                <li><a href="#">Teachers</a></li>
                <li><a href="#">Rooms</a></li>
            </ul>
        </div>

        <div class="footer-col">
            <h4>Resources</h4>
            <ul>
                <li><a href="#">Research Labs</a></li>
                <li><a href="#">Publications</a></li>
                <li><a href="#">Student Portal</a></li>
                <li><a href="#">Support</a></li>

        </div>

        <div class="footer-col">
            <h4>Newsletter</h4>
            <p>Stay updated with our latest news.</p>
            <form class="newsletter">
                <input type="email" placeholder="Your email">

        </div>

    </div>

    </div>
</footer>

</body>
</html>
In this base template, we define a header, a main content area defined by the {% block content %} tag, and a footer.
Now, let's modify the index.html, News.html, About_Us.html, and Contact_Us.html template to extend the base template and add specific content to the content block: 
{% extends "Home/base_Home.html" %}
{% block content %}
<h1>Home</h1>
{% endblock %}
In this modified index.html template, we use the {% extends %} tag to specify that this template extends the base_Home.html template. We then define the content for the content block using the {% block content %} and {% endblock %} tags.
You can similarly modify the News.html, About_Us.html, and Contact_Us.html templates to extend the base template and add specific content to the content block.
This block, named content, is a placeholder; the child template will define the kind of information that goes in the content block.
Everything that we aren't inheriting from the parent template goes inside a content block.
When Django processes this template, it will combine the base template with the content defined in the content block.
A child template doesn't have to define every block from its parent, so you can reserve space in parent templates for as many blocks as you like, and the child template uses only as many as it requires.
The code used in templates differs from Python in some important ways. Python uses indentation to indicate which lines of a for statement are part of a loop. In a template, every for loop needs an explicit {% endfor %} tag indicating where the end of the loop occurs.
To print a variable in a template, wrap the variable name in double curly braces ({{ and }}).

Styling with CSS

To style your Django web pages, you can use CSS (Cascading Style Sheets).
Create a new directory called static inside the Home app directory, and then create another directory called Home inside the static directory. This structure helps Django locate the static files for the Home app.
Inside the Home/static/Home directory, create a new CSS folder then create a file called styles.css and add your CSS styles to it.
To link the CSS file to your HTML templates, you need to load the static files in your base template (base_Home.html) and include a link to the CSS file in the head section: 
{% load static %}
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <link rel="stylesheet" href="{% static 'Home/CSS/styles.css' %}">
    <title>University</title>
    
</head>
<body>
In this code, we use the {% load static %} tag to load the static files and the {% static 'Home/CSS/styles.css' %} tag to generate the URL for the styles.css file.
Now, when you run the development server and navigate to your web pages, the styles defined in styles.css will be applied.
If it does not work, make sure that the STATIC_URL setting in settings.py is set correctly: 
STATIC_URL = '/static/'
This setting tells Django where to look for static files. By default, it is set to /static/, which matches the directory structure we created earlier.
Or restart the server to see the changes.
